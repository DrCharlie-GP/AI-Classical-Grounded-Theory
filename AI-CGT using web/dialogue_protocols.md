# 对话协议

## 总体原则

每一次与 LLM 的对话应有**明确的单一任务**。不要开一个对话横跨多个任务——那会让 LLM 的 context 混乱，也让研究者自己的判断在多任务切换中被稀释。

每次对话开始时，**先粘贴对应类型的开场指令**，让 LLM 知道本次对话的性质与边界。开场指令是研究者给 LLM 的"角色约束"，同时也是研究者自己的"纪律提醒"。

每次对话结束前，**粘贴对应类型的收尾指令**让 LLM 做封存。封存的产出由研究者保存到本地 `dialogue_logs/` 目录，按 `dialogue_log_TEMPLATE.md` 的格式归档。

本文件定义四类对话协议。研究者根据需要选择。不在这四类中的对话——例如闲聊、让 LLM 协助写非研究相关的邮件——不应在本 Project 内进行，应另开普通 Claude 对话。

---

## 类型 A：方法论咨询对话

### 用途

研究者遇到 CGT 方法论上的疑问，例如"在中心范畴浮现后还能不能增加新的受访者"、"备忘录在什么情况下应该被合并"、"我的当前做法是否违反了 SKILL 的某一条禁止"。LLM 在此类对话中的角色是"方法论文档的解释者"，回答严格基于 Project Knowledge 中的 SKILL 与 REFERENCE，不做自由发挥。

### 使用时机

- 一周中随时，但建议集中在周五下午整理阶段。
- 新手阶段（研究前 4-6 周）可能较频繁（每周 2-3 次），研究成熟后频率降低。

### 不适用于

- 对资料本身的判断问题（那属于类型 B 或 C）。
- 关于研究进展的整体诊断（那属于类型 D）。
- 关于某个概念是否合适、是否应该调整的判断——这类问题永远由研究者自己回答，不应向 LLM 咨询。

### 开场指令（复制粘贴）

```
本次对话类型：A（方法论咨询）。

请严格基于 Project Knowledge 中的 classical_grounded_theory_SKILL.md 与
classical_grounded_theory_REFERENCE.md 回答我的方法论问题。回答要求：

1. 明确指出答案来自 SKILL 的哪一节或 REFERENCE 的哪一节，附条款编号。
2. 如 SKILL 与 REFERENCE 之间存在表述差异，以 Glaser 原著引文为准。
3. 如我的问题所涉内容在两份文件中都没有直接对应，明确告诉我"文件中
   没有直接对应条款，以下是我基于方法论气质的推断"，并将推断与文件
   内容严格分开。
4. 不要给出"平衡的双方观点"——CGT 在方法论上是有立场的。如我询问的
   问题涉及 CGT 与其他版本（Strauss & Corbin 或 Charmaz）的冲突，请
   明确给出 CGT 的立场。
5. 不要过度解释。简短、准确、带引证。

我的问题是：[在此填写问题]
```

### 收尾指令（复制粘贴）

```
请用不超过 100 字总结本次对话的结论（不是你说了什么，而是我可以带走
的一条或几条明确判断）。同时列出本次对话引用到的 SKILL 或 REFERENCE
的具体条款编号，供我日后追溯。
```

### 产出处理

收尾指令的输出复制到本地 `dialogue_logs/YYYY-MM-DD_A_methodology_[主题].md`，按 `dialogue_log_TEMPLATE.md` 格式归档。**此类对话的产出通常不上传 Project Knowledge**——因为它是对已有文档的解释，不增加新研究材料。

## 类型 B：不断比较素材对话

### 用途

研究者已经浮现出一个概念（例如"向下比较"），想把已标记为该概念的所有资料片段并列呈现，做事件—事件或概念—更多事件的比较。LLM 在此类对话中的角色是"把资料摆好的助手"——并列展示，**不做相似度排序、不做自动归纳、不做概念推荐**。

### 使用时机

- 当某个概念的相关片段累积到 5 份以上时。
- 在进入选择性编码阶段前，用于验证中心范畴候选。
- 不频繁——大概每周 1-2 次，每次聚焦一个概念。

### 不适用于

- 当概念还不清晰、尚未命名时——那是研究者独自浸润的工作。
- 当目标是"让 LLM 帮我发现新概念"时——那违反 SKILL §6-第 2 条。

### 准备工作

对话前，研究者从编码数据库（或本地笔记）中检索所有被标记为该概念的资料片段，复制到一份文件中，每段前标注 source_ref（例如 `P01-第 17 段`）。对话时把这份文件内容粘贴给 LLM。

### 开场指令（复制粘贴）

```
本次对话类型：B（不断比较素材）。

我将粘贴一组已被我标记为同一概念的资料片段，每段前有来源标注。
概念名是：[填入概念名]。

请你做的事：

1. 把这些片段按来源标注并列展示，每段保持原文字不变。
2. 如我要求，从中摘出包含某个关键词的片段子集。
3. 如我要求，检查片段间是否存在我可能忽略的表述差异（**仅指出
   差异，不做解释**，解释由我自己做）。
4. 如我要求，接受我粘贴额外的"反驳性资料"（来自其他访谈中 NOT
   被标记为该概念的片段），并把它们与主集并列展示。

请你绝对不做的事：

1. 不做相似度排序或聚类。
2. 不对这些片段做归纳或抽象总结。
3. 不推荐新概念、不评价现有概念是否合适。
4. 不对资料做"常识性解读"——研究者解读是我的工作。

准备好后告诉我，我开始粘贴片段。
```

### 收尾指令（复制粘贴）

```
本次对话结束前，请用不超过 150 字总结**我（研究者）在本次对话中做
出的新判断**。不是总结你说了什么，是总结我基于比较形成的新认识。
如我本次没有做出新判断，就直接说"本次对话无新判断"。不要编造。

然后另起一段，列出本次对话中涉及的所有 source_ref，作为日后追溯的
索引。
```

### 产出处理

收尾指令的输出复制到本地 `dialogue_logs/YYYY-MM-DD_B_comparison_[概念名].md`。如对话中研究者形成了新的理论想法，应立即手写新备忘录，**独立于 LLM 对话输出**——LLM 产出只作为起点，备忘录是研究者自己的概念化产物。

如果本次对话形成的新判断重要到值得全项目知晓（例如浮现了中心范畴候选），把对话日志上传到 Project Knowledge，供后续对话可引用。

## 类型 C：备忘录写作陪练

### 用途

研究者已经在本地写好一份备忘录草稿，想让 LLM 做**语言层面**的质询——指出表述不清、同词异义、结构不连贯等问题。LLM 的角色是"语言顾问"，不是"概念顾问"。

### 使用时机

- 完成备忘录初稿后、存入 `memos/by-category/` 前。
- 每份备忘录最多使用一次——避免反复打磨单篇备忘录而耽误研究节奏。
- Glaser（1978）强调备忘录是"研究者捕捉当下想法"的文字，过度打磨会失去那种临场性。本协议的作用是纠正明显的语言问题，不是让备忘录变"漂亮"。

### 不适用于

- 请 LLM 评价"这个概念是否合适"——那是研究者自己的工作。
- 请 LLM 补充"你漏了 X 方面"——那是让 LLM 的预设介入研究者的概念化，违反 SKILL §6-第 2 条。
- 正式论文章节的写作——那属于写作阶段，另用其他对话类型。

### 开场指令（复制粘贴）

```
本次对话类型：C（备忘录写作陪练）。

我将粘贴一份备忘录草稿。你的任务严格限定为两项语言层面的质询：

1. 指出我表述中不清晰、歧义、或自相矛盾的地方。对每一处，引用我
   的原文，指出问题，不给修改建议（修改由我自己做）。
2. 指出我在同一份备忘录中用同一个词指代不同含义的地方，或用不同
   的词指代同一个概念的地方。同样只指出，不给修改建议。

你绝对不做的事：

1. 不评价我的概念是否合适、是否成立、是否符合某理论。
2. 不建议新概念、不推荐相关文献、不补充"你漏了 X"。
3. 不重写我的文字。
4. 不对我的备忘录做整体评价（如"这份备忘录很有深度"）——不需要。
5. 不把我的表述与任何现有理论或文献对接。

准备好后告诉我，我粘贴草稿。
```

### 收尾指令（复制粘贴）

```
本次对话结束前，请生成两样东西：

1. 我在本次对话中决定接受的修订点的清单（不是你建议的，而是我明
   确说"好，改"的那些）。
2. 我决定保留原状的点的清单（你指出了问题但我认为原状更好）——
   这一项对后续反思很重要。

仅此两项，不要追加建议。
```

### 产出处理

根据对话结果修订备忘录后，将修订版保存到 `memos/by-category/`。对话日志存入 `dialogue_logs/YYYY-MM-DD_C_memo_[范畴名].md`。**此类对话的产出不上传 Project Knowledge**——备忘录本身也不上传（备忘录只在需要时按具体对话粘贴）。

## 类型 D：阶段性反思对话

### 用途

每 2-3 周一次的大型反思对话。研究者把当前的 `research_state.md` 与本阶段新增的备忘录目录交给 LLM，让它按 SKILL §8 的九类红旗做诊断，按 REFERENCE §3 的中心范畴判据评估候选，指出研究者可能已经发生但没察觉的方法论越界。

LLM 在此类对话中的角色是"严格的方法论审稿人"。它的意见**不是结论**，是研究者需要反思的素材。

### 使用时机

- 每 2-3 周一次，固定安排（例如每月第一个和第三个周五）。
- 在重要的研究节点前：进入选择性编码前、理论饱和判定前、外部比较阶段前、开始正式写作前。

### 不适用于

- 频繁使用。此类对话的价值在于间隔——间隔让研究者有时间独立消化前一次的反思结果。每周都做会让研究者陷入"被 LLM 驱动"的模式。

### 准备工作

对话前确保 `research_state.md` 已更新到最新版本并上传到 Project Knowledge。准备好本阶段新增的所有备忘录文件名清单。

### 开场指令（复制粘贴）

```
本次对话类型：D（阶段性反思）。

我已更新 research_state.md 到 Project Knowledge。请先基于该文件回答
以下问题，然后依次进入诊断环节。

第一步：请基于 research_state.md 回答：

1. 我当前处于哪个研究阶段？
2. 根据该阶段，SKILL §9 工作流模板中对应的活动有哪些？
3. 我在 research_state.md 中呈现的工作与上述活动是否对齐？

回答完第一步后等我确认再继续。

第二步：按 SKILL §8 的九类红旗（A 到 I）逐一对照我的当前研究状态。
对每一类红旗：

- 若无越界风险：直接说"A 无风险"。
- 若可能有越界：引用 research_state.md 中的具体表述，说明为何可能触发该红旗，并引用 SKILL 的具体条款。

重要：你对"可能越界"的判定只是"值得我反思"的信号，不是结论。你
不需要过度保守（把所有模糊都标红），也不需要过度宽松（掩盖真实
风险）。

第三步：按 REFERENCE §3 的十一项判据评估我在 research_state.md §4
中列出的每一个中心范畴候选。对每一个候选：

- 对每一项判据：当前证据是否已满足？满足到什么程度？
- 综合：此候选是否已足够支持中心范畴的判定？有何不足？

重要：你不推荐最终选择哪一个候选为中心——那是我的工作。你只评估
每个候选在各判据上的当前状态。

第四步（可选）：若上述三步中出现我没主动提到、但你认为值得我反思
的问题，提出来。同样以"值得反思的问题"形式，不以"我建议"形式。

请从第一步开始。
```

### 收尾指令（复制粘贴）

```
本次反思对话结束前，请生成一份结构化诊断报告，包含四节：

1. **当前阶段与活动的对齐情况**（1 段）
2. **SKILL §8 九类红旗的诊断结果**（每一类一行，无风险的写"无风险"，
    有风险的引用具体证据与 SKILL 条款）
3. **中心范畴候选在 REFERENCE §3 判据上的当前状态**（每个候选一个
    小表，列出每一判据的满足程度）
4. **其他值得研究者反思的问题**（若有）

报告使用客观、简洁的语言，不加入你的立场表达（如"我建议"、"我认为
你应该"）。报告将作为研究者的反思素材，由研究者自行决定哪些采纳。

报告结束后另起一段，写一句话：这份报告不是研究结论，是研究者反思
的素材。
```

### 产出处理

诊断报告保存为 `logs/reflection_YYYY-MM-DD.md`。研究者**在接下来的 2-3 天内**独立消化报告，不立即行动。消化完成后在 `logs/methodological_memos.md` 中写一份"对本次诊断的回应"——记录哪些采纳、哪些不采纳、各自的理由。

诊断报告本身以及研究者的回应都上传到 Project Knowledge，供下次类型 D 对话的连续性使用。这是唯一需要把对话产出上传 Project Knowledge 的情况。

## 常见问题

**问：我可以把四种类型的开场指令合并为一个"通用开场"吗？**
不建议。通用指令会模糊每次对话的目标，让 LLM 的回应变得折中、平均、无锋。分类型的指令迫使研究者每次对话前做一次判断——"我这次要做什么"——这个判断本身是对 CGT 纪律的维护。

**问：如果一次对话中我想切换类型怎么办？**
关闭当前对话，开新对话。Claude Project 中的对话是便宜的——开新对话的成本远低于在一个对话中混合任务带来的 context 污染。

**问：LLM 如果没有严格遵守我开场指令中的限制怎么办？**
立即指出。例如它在类型 B 中开始归纳片段，你回复："请停下。按开场指令，你不做归纳。请回到并列展示。"——LLM 会立即纠正。如果它在同一次对话中多次越界，关闭对话重新开始。

**问：所有类型的对话都要粘贴收尾指令吗？**
是的。收尾指令的作用不仅是产出总结，更是迫使 LLM 在结束前"梳理自己说了什么"——这减少了研究者把 LLM 的推测性表述误记为确定判断的风险。

**问：收尾指令的产出我都要归档吗？**
是的，即使对话内容简单。归档的目的不是事后翻阅，是**研究过程透明性**。答辩时如果有人问"你是如何形成这个概念的"，你能拿出时间序列的对话日志证明每一步的思考都有据可查。这是 CGT 研究在 AI 时代保护自己的关键。
